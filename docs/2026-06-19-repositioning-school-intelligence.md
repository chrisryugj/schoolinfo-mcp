# 개편 계획 — "매일 일상보조" → "학교선택·수행평가 인텔리전스"

작성: 2026-06-19 · 대상: schoolinfo-mcp (v0.2.0, web school-mcp.fly.dev)
상태: **제안 (실행 전 9장 결정 확인 필요)**

## 1. 배경 — 받은 피드백과 진단

공개 후 받은 외부 피드백 3건 — **전원 현직 교사·교육현장** — 의 핵심:

- **(교사) "무엇을·누구를 위한 도구인지 빠졌다."** 급식은 매일 갱신돼도 중·고 학부모가 굳이 찾지 않고, 학사일정은 학기초 1회 공지로 끝, 수행평가만 수요가 보인다.
- **(교사) "이미 점유된 시장."** 재학생 학부모는 e알리미(안내장·급식·일정·시험)·하이클래스·리로스쿨(학급 알림)·NEIS(행정)로 정보를 **푸시**받고 있다. 급식 메뉴는 교과서 출판사들이 학교별 API로 이미 깔아놨다.
- **(교사) "수행평가·정기고사 = 수요 확실. 주변 학교 비교 = 매우 유의미."** 급식·학사일정은 리로스쿨/하이클래스 대체 불가(인정). 대신 교사 본인이 **"인근/전체 고등학교 평가계획을 다운받아 수행평가 반영비율·형식을 비교"**하고 싶다고 — 현재 도구에 없는 빈칸을 정확히 지목.

> **출처가 전원 교사라는 점이 두 가지로 작용한다.** ① 신호의 무게 — "일상정보는 대체 불가, 수행평가·정기고사만 수요"가 외부인 추측이 아니라 현장 교사의 일관된 판단이다(일상정보 격하·B축 강화의 근거가 단단해짐). ② **축 C가 가설→실증** — 자발적으로 디테일한 피드백을 준 초기 반응자가 전부 교사 = 교사가 실제 첫 사용자층일 수 있다(GTM은 교사 커뮤니티부터). 단, *제공자 네트워크가 교사 편향*일 수 있어 학부모축(A/B)의 시장 수요는 별도 검증 대상이다. → **이 빈칸이 채워졌다: §11(2026-06-20) 첫 학부모 신호 도착.**

**진단**: 현재 제품의 1순위 셀링포인트(웹 hero "오늘 급식부터…", `get_school_meal`/`get_school_week`/`get_parent_digest`)는 전부 **재학생 일상정보** = 이미 e알리미가 푸시로 점유한 전장이다. MCP/웹은 사용자가 물어봐야 답하는 **pull**이라 푸시 incumbent를 못 이긴다. 피드백은 옳다.

**그러나** 현재 도구 13종 안에는 incumbent가 **못 채우는 빈틈**을 정확히 겨냥한 기능이 이미 들어있다 — `get_school_report`(학교 비교), `get_evaluation_plan`(hwp 속 수행평가). 문제는 코드가 아니라 **포지셔닝**이다. 잘못된 전장을 1순위로 광고하고 있다.

## 2. 전략 — 전장을 옮긴다 (세 모집단)

"재학생 일상보조"를 1순위에서 내리고, incumbent가 비워둔 빈틈으로 간판을 바꾼다. **"재학생"을 버리는 게 아니라 "일상정보"를 버린다.** 교사 피드백으로 모집단이 하나 더 열렸다(축 C).

| 축 | 모집단 | 통증 (incumbent가 못 주는 것) | 핵심 도구 |
|----|--------|------------------------------|-----------|
| **A. 학교 고르기** | 예비/이사/배정 고민 학부모 — **아직 그 학교 계정이 없음** | e알리미는 재학생 계정 전용 → 비교 불가. 여러 학교를 한 표로 비교할 데가 없다 | `get_school_report`, `get_disclosure`, `get_subject_achievement` |
| **B. 내신·입시** | 재학생 학부모 | **수행평가·정기고사 반영비율·평가기준**이 hwp 첨부에 숨어있어 못 본다. e알리미도 안 풀어줌 | `get_evaluation_plan`, `get_admission_subjects`, `get_subject_achievement` |
| **C. 다학교 평가계획 비교 (신규)** | **교사** | 인근/전체 학교의 수행평가 반영비율·형식을 **교차 비교**할 데가 없다. e알리미·리로스쿨은 학부모↔학교 1:1용이라 교차 분석 불가 | **신규** `compare_evaluation_plans`(가칭) = `get_evaluation_plan` 다학교 확장 |

> **B축은 "수행평가"만이 아니다** — 지필(정기고사)↔수행 반영비율 등 **내신 평가구조 전체**다. 교사가 "정기고사도 수요 확실"이라 명시했고, 평가계획 문서 안에 이 비율이 함께 들어있다.

**왜 이 세 곳은 이기나**: A는 incumbent의 구조적 사각(계정 없는 사람 배제), B는 incumbent가 파일을 그냥 던질 뿐 파싱·구조화를 안 함, C는 *교차 학교 분석*을 아무도 안 함(전부 단일 학교-학부모 소통용). 셋 다 "흩어진 공공데이터를 쓸 수 있게 가공"이라는 이 도구의 원래 강점과 정렬된다.

## 5. 웹앱 UX 개편 (`src/web.ts`)

현재: 모드탭 2개([🏫 우리 학교][🎓 대학 진학]), hero가 급식 우선.

개편안 — 모드탭 3개로 재편하고 진입 동선을 두 축으로:

```
[📊 학교 비교]  [📋 수행평가·내신]  [🎓 대학 진학]
```

- **📊 학교 비교 (신규 전면)**: 지역·학교급 선택 → `get_school_report` 표를 첫 화면에. "우리 동네 중학교 한눈에 비교". 내 학교 행 하이라이트(이미 `.mine` 스타일 있음 → 재활용).
- **📋 수행평가·내신**: 학교명 검색 → `get_evaluation_plan` 표 + `get_subject_achievement` 딥링크. 현재 "우리 학교" 탭의 수행평가 부분을 독립.
- **🎓 대학 진학**: `get_admission_subjects`. 현재 유지.
- **급식·학사일정·이번주**: 모드탭에서 빼고, 학교 검색 후 **상세 화면 하단 접이식**("📅 학사일정 / 🍱 급식 더보기")으로 격하. 코드 재사용, 위치만 이동.

hero 카피: §6 참조. 기존 2열 요약 박스(`.핵심 두 줄 요약 수행평가/급식`)는 **수행평가 / 학교비교**로 교체.

> **교사축(C)은 학부모 모드탭과 섞지 않는다.** 사용 맥락이 달라(교사의 교차 학교 분석 vs 학부모의 내 학교) 한 화면에 합치면 둘 다 흐려진다. 별도 진입(예: 푸터 "교사용 평가계획 비교")으로 두고 Phase 6에서 형태 확정.

## 7. 실행 Phase

작은 것부터, 되돌리기 쉬운 순서로. 각 Phase 끝에 빌드+배포로 검증.

- **Phase 0 — 합의** (코드 0): 본 문서 §9 결정 확정.
- **Phase 1 — 카피/메타만** (저위험, 즉시 효과): §6 표 적용. title/h1/og/package desc. 기능 변경 없음 → 배포해서 공유 카드·검색 노출부터 바꾼다. **여기서 멈춰도 포지셔닝 절반은 달성.**
- **Phase 2 — 모드탭 재편**: web.ts 모드탭 2→3개. 📊 학교 비교를 신규 추가하고 기본 진입으로. 급식/일정을 상세 하단 접이식으로 이동(코드 재사용).
- **Phase 3 — 비교 화면 강화**: `get_school_report` 결과에 "내 학교 행 하이라이트", 정렬, 지표 설명 툴팁. 비교가 첫인상이 되도록 다듬기.
- **Phase 4 — 수행평가 심화**: `get_evaluation_plan` + `get_subject_achievement` + `get_admission_subjects`를 하나의 "내신→입시" 흐름으로 연결(과목 선택 → 권장 이수과목 제안).
- **Phase 5 — 정리**: CHANGELOG, README 갱신. `neis-parent-killer-features.md` 계획은 deprecated 표기(이 문서가 대체).
- **Phase 6 — (교사축) 다학교 평가계획 비교**: `get_evaluation_plan`을 N개 학교로 확장 → 수행/지필 반영비율·평가형식을 한 매트릭스로. **별도 트랙**(학부모 Phase 1~5와 독립, 후순위). 핵심 난도 = hwp 표 포맷이 학교마다 달라 비율 숫자 정규화가 어려움 → **소수 학교 PoC로 추출 신뢰도부터 검증** 후 확대. 교사 커뮤니티가 학부모와 별개 distribution이 될 수 있어 가치 큼.

권장: **Phase 1만 먼저 배포 → 일주일 반응 보고** Phase 2 진행. 카피 교체는 1시간, 효과 검증 가치 큼. Phase 6(교사축)은 A/B 안정화 후 PoC로 분리 착수.

## 8. 검증·성공기준

- 기능 회귀 없음: `npm test` (admission/evaluation/meal/schedule 테스트 그대로 통과 — 코드 삭제 안 하므로 깨질 일 없음).
- 빌드: `npm run build` → fly 배포.
- 포지셔닝 체크(정성): 처음 보는 사람에게 hero만 보여주고 "이게 누구를 위한 거 같아?" → "학교 고르거나 내신 챙기는 학부모"가 나오면 성공. 기존엔 "급식 보는 앱"이 나왔을 것.
- (선택) 모드탭별 사용 로그 — 어느 축에 실수요가 붙는지 측정 후 더 투자.

## 9. 열린 결정 (실행 전 확인)

1. **급식·학사일정 = 격하(접이식) vs 완전 삭제?** → 추천: **격하**. 삭제는 이미 만든 자산 폐기 + 회귀 리스크, 효과는 격하와 동일(노출만 내리면 됨).
2. **모드탭 2개 vs 3개?** → 추천: **3개**(비교/수행평가/진학). 단 "수행평가"와 "진학"이 가까워 2개([학교 비교][내신·진학])로 합치는 안도 가능.
3. **Phase 1만 먼저 vs 한 번에?** → 추천: **Phase 1 선배포** 후 반응 보고 결정.
4. 컨퍼런스 데모(간판)와 충돌 없는지 — 이 개편이 데모 시나리오를 바꾸는지 확인 필요.
5. **교사축(C, Phase 6) 우선순위** — A/B(학부모)를 먼저 굳히고 PoC로 분리할지, 교사 수요가 더 뾰족하니 먼저 칠지. **초기 반응자가 전원 교사**인 점은 C 우선 착수 쪽 근거(단 표본편향 감안). 추천: 택일 아니라 **트랙 분리** — A/B는 즉시 효과·저위험이라 바로 가고, C는 교사가 첫 사용자층일 수 있으니 **병렬로 소수 학교 PoC 착수**(hwp 정규화 신뢰도를 일찍 검증).

## 11. 학부모 가독성 피드백 (2026-06-20 — 첫 외부 학부모 신호)

§1~10의 피드백 3건은 전원 교사였다. 이번 건은 **1촌이 아니라 댓글로 의견을 남긴 초등학생 아버지(재학 초등 학부모)**(고영혁 공유 경로) — §1에서 "학부모축(A/B) 시장수요는 별도 검증 대상"이라 비워둔 칸의 **첫 자발적 학부모 신호**다.

> **출처가 "초등"이라는 점을 먼저 못 박는다.** 이 신호는 **B축(내신·입시)을 입증하지 않는다** — 초등은 정기고사·내신·수행평가 반영비율이 없어 B축의 핵심 셀링포인트(평가 반영비율 비교) 타겟이 아니다. 초등 학부모가 본 "평가계획"은 *내신 비교*가 아니라 *우리 아이가 뭘 배우고 어떻게 평가받나*의 **열람**이다. 따라서 아래 #1 개선의 우선순위 근거는 "B축 직결"이 **아니라** "재학 학부모의 정보 열람 가독성"이다. (B축 실타겟인 **중·고 학부모 신호는 여전히 미확인** — 표본은 초등 1명.)

**전략적 의미:**
- **(a) §2 "재학생이 아니라 *일상정보*를 버린다"의 실증.** 재학 초등 학부모인데도 거론한 건 평가계획·공시(학생수)·학사일정 셋뿐 — **급식 등 일상정보 언급 0.** 급식을 가장 많이 볼 법한 초등 학부모조차 일상정보가 아니라 *가공된 공시*에서 가치를 느꼈다("흩어져 불편했는데 모여 있어 좋다"). = 일상정보 격하(§4)는 맞고, **재학생 층 자체는 버릴 대상이 아니라는** 두 번째 모집단발 증거. (단 표본 1, 초등 1케이스.)
- **(b) 통증이 "기능 부재"가 아니라 "가독성".** 셋 다 *기능은 있는데 읽기 불편하다*다. 사용자가 원한 기능은 이미 있으니 **다음 레버는 렌더링 다듬기**임을 가리킨다. → 신규 Phase 불필요, 기존 Phase 3·4에 흡수.

**3건 → 기존 Phase 매핑 (코드 조사로 현재 상태 실측):**

| # | 학부모 통증 | 현재 상태 (실측) | 조치 | 흡수 Phase |
|---|------------|-----------------|------|-----------|
| 1 | 평가계획이 교과별로 구분 안 됨. 제목이 날아감. 교과별로 접어두고 원하는 것만 펼치고 싶다 | 렌더 3갈래: `structured`(통합형→과목칩 아코디언·해결), `list`(과목별 파일→과목 버튼·제목 보존·해결), **`else`=plain 마크다운 fallback([web.ts:921](../src/web.ts#L921))→교과 구분·접힘 둘 다 없음**. 학부모 학교는 fallback 케이스로 추정(본인도 "양식 다른지 모르겠다"고 정확히 의심) | **(재현 먼저)** 해당 학교 양식으로 어느 갈래로 떨어지는지 확인 → fallback 마크다운에도 **교과 제목 헤더 추출 + `<details>` 기본 접힘** 적용. 이미 동작하는 `structured` 경험을 fallback까지 끌어올림 | **Phase 4** (수행평가 심화) — 단 *근거는 B축 내신비교가 아니라 재학 학부모 열람 가독성*. fallback 보강은 중·고 B축에도 그대로 이득이라 우선순위 높음 |
| 2 | 학년별·학급별 학생수가 순서·구분 없이 섞임 | 다학교 비교표(`renderCompare`)는 학년 열 분리·학생수순 정렬 됨. **단일 학교 공시 상세**에서 학년/학급 순서·구분 약함. **학급 단위 데이터가 공시 API에 실제 존재하는지 미확인** | **(데이터 확인 먼저)** 학급별 수치가 공시에 있으면 학년>학급 계층 표로, 없으면 **학년 오름차순 정렬·구분 강화로 한정**(없는 데이터 지어내지 않음) | **Phase 3** (비교 화면 강화) — A축 보강 |
| 3 | 방학처럼 긴 일정은 x월 xx일 ~ x월 xx일로 모아서 | 시험만 `groupExamPeriods`로 범위 병합. **방학 등 연속 일반일정은 개별 날짜로 나열**(renderSchedule) | renderSchedule에 **연속 동일 일정 범위 병합** 추가(시험 로직 재사용, 저비용) | **상세 하단 보조정보**(§5) — 학사일정은 Demote 대상이라 **최저 우선순위**, "보조정보 다듬기" 수준 |

**우선순위:** #1(재학 학부모 열람 가독성 + fallback 보강이 중·고 B축에도 그대로 이득) ≫ #2(A축 보강) ≫ #3(Demote 정보, 저비용이면 끼워넣기). 셋 다 **기능 추가 아닌 가독성 개선**이라 회귀 리스크 낮음.

> **단정 주의:** #1은 "이미 해결"이 아니라 "**구조화 파이프를 타는 학교만 해결**". #2는 학급 데이터 존재 여부가 미확인. 둘 다 **재현·데이터 확인을 조치 1단계로** 둔다(없는 걸 만들지 않기 위해).
